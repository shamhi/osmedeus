# 3. DIRECTORY & FILE DISCOVERY
# ═══════════════════════════════════════════════════════════════════════════════

## 3.1 Tool Selection

```
Tool         | Best For                      | Speed  | Depth
-------------|-------------------------------|--------|--------
ffuf         | Flexible fuzzing, fast        | ★★★★★  | ★★★
feroxbuster  | Recursive discovery           | ★★★★   | ★★★★★
gobuster     | Simple directory brute-force  | ★★★★   | ★★★
dirsearch    | Good defaults, easy to use    | ★★★    | ★★★
```

## 3.2 Wordlist Strategy

Choose wordlists based on the identified technology:

```
Technology    | Wordlist Priority
--------------|----------------------------------------------------
Generic       | raft-medium-directories, common.txt, dirsearch default
PHP           | PHP-specific paths, .php extensions, common CMS paths
Java/Spring   | /actuator/*, /swagger-ui/, /api-docs, /WEB-INF/
.NET          | /web.config, /elmah.axd, /trace.axd, /_vti_*
Node.js       | /api/, /graphql, /socket.io/, package.json endpoints
Python/Django | /admin/, /static/, /media/, __pycache__
Ruby/Rails    | /rails/info/routes, /assets/, /sidekiq
WordPress     | /wp-admin/, /wp-content/, /wp-includes/, xmlrpc.php
```

## 3.3 Intelligent Discovery Patterns

```
Step 1: Quick check for low-hanging fruit
  - robots.txt, sitemap.xml, .well-known/*
  - Common sensitive files: .env, .git/HEAD, .DS_Store, backup.zip
  - Admin panels: /admin, /login, /dashboard, /console

Step 2: Technology-specific paths
  - Based on identified framework/CMS
  - Version-specific paths (e.g., Spring Boot actuators changed endpoints
    between v1 and v2)

Step 3: Extension fuzzing
  - Try discovered paths with multiple extensions:
    .bak, .old, .orig, .swp, .txt, .log, .conf, .xml, .json, .yaml

Step 4: Recursive exploration
  - For each discovered directory, fuzz one level deeper
  - Focus on directories that return interesting status codes

Step 5: Backup and archive hunting
  - [filename].bak, [filename]~, [filename].old
  - [domain].zip, [domain].tar.gz, backup.sql
  - Source code archives: .git/, .svn/, .hg/
```

## 3.4 Status Code Interpretation

```
Code | Meaning                           | Action
-----|-----------------------------------|----------------------------------
200  | Found — examine content           | Analyze response body
201  | Created — potentially interesting  | Check for creation side effects
301  | Permanent redirect                | Follow and test destination
302  | Temporary redirect                | Note — may indicate auth redirect
307  | Temp redirect preserving method   | Important for POST endpoints
401  | Auth required — something exists  | Try auth bypass techniques
403  | Forbidden — definitely exists     | Try 403 bypass techniques
405  | Method not allowed                | Try other HTTP methods
500  | Server error — possible injection | Test for error-based vulns
```

# 4. PARAMETER FUZZING & INJECTION
# ═══════════════════════════════════════════════════════════════════════════════

## 4.1 Parameter Discovery

Before injecting, discover all parameters:

```
Sources:
  1. URL query parameters (visible in links/forms)
  2. POST body parameters (form submissions)
  3. Hidden form fields (view HTML source)
  4. JSON request bodies (API calls)
  5. HTTP headers (custom headers, cookies)
  6. URL path parameters (/api/users/{id})
  7. Parameter mining tools (Arjun, paramspider)
  8. JavaScript source analysis (grep for param names)
  9. API documentation (Swagger, GraphQL introspection)
```

## 4.2 Injection Test Matrix

For each discovered parameter, test systematically:

```
Vulnerability  | Test Payloads                         | Detection
---------------|---------------------------------------|------------------
SQL Injection  | ' OR 1=1--                            | Error/behavior change
               | ' AND SLEEP(5)--                      | Time delay
               | ' UNION SELECT NULL,NULL--            | Column count
XSS            | <script>alert(1)</script>             | Reflected in response
               | "><img src=x onerror=alert(1)>        | HTML context escape
               | javascript:alert(1)                   | URL/href context
               | {{constructor.constructor('alert(1)')()}} | Template injection
SSTI           | {{7*7}} → 49                          | Math evaluation
               | ${7*7}                                | Alternative syntax
               | <%=7*7%>                              | ERB syntax
Command Inj.   | ;id                                   | Command output
               | |whoami                               | Piped command
               | $(cat /etc/passwd)                     | Subshell
Path Traversal | ../../../etc/passwd                    | File contents
               | ....//....//etc/passwd                 | Double encoding
               | ..%252f..%252f                         | Double URL encoding
SSRF           | http://127.0.0.1                      | Internal access
               | http://169.254.169.254/                | Cloud metadata
               | http://[::1]                          | IPv6 bypass
LDAP Inj.      | *)(objectClass=*                      | LDAP response change
XXE            | <!DOCTYPE x [<!ENTITY e SYSTEM "file:///etc/passwd">]> | File read
```

## 4.3 Context-Aware Injection

Adapt payloads based on where input is reflected:

```
Context                    | Escape Sequence           | Payload
---------------------------|---------------------------|------------------
HTML body                  | <tag>                     | <img src=x onerror=...>
HTML attribute (quoted)    | ">                        | "><script>...</script>
HTML attribute (unquoted)  | space                     | onmouseover=alert(1)
JavaScript string          | ';                        | ';alert(1)//
JavaScript template lit.   | ${                        | ${alert(1)}
URL parameter              | javascript:               | javascript:alert(1)
CSS value                  | }                         | };background:url(...)
JSON value                 | "                         | ","injected":"value
HTTP header                | \r\n                      | Header injection (CRLF)
```

# 5. AUTHENTICATION & AUTHORIZATION TESTING
# ═══════════════════════════════════════════════════════════════════════════════

## 5.1 Authentication Bypass Techniques

```
Technique                        | Method
---------------------------------|----------------------------------------
Default credentials              | admin:admin, admin:password, test:test
SQL injection in login           | admin' OR '1'='1' --
Rate limit bypass                | IP rotation headers, case variation
Account enumeration              | Different errors for valid vs invalid users
Password reset flaws             | Predictable tokens, no expiration
JWT attacks                      | alg:none, HS256→RS256 confusion,
                                 | weak secret brute-force (jwt_tool)
Session fixation                 | Pre-set session ID before auth
Cookie manipulation              | Decode and modify session cookies
OAuth flaws                      | Open redirect in redirect_uri,
                                 | missing state parameter
2FA bypass                       | Direct access to post-2FA endpoints,
                                 | brute-force short codes, backup codes
```

## 5.2 Authorization Testing (Access Control)

```
Test Type           | Method
--------------------|----------------------------------------------------
Vertical escalation | Access admin endpoints with regular user token
Horizontal IDOR     | Replace user ID in API call (/api/users/123 → 124)
Method-based bypass | Change GET to POST/PUT/PATCH/DELETE
Path traversal      | /admin/../user/settings → /admin/settings?
Parameter pollution | ?role=admin, ?isAdmin=true added to requests
Missing auth check  | Remove auth header entirely — still works?
GraphQL authz       | Query other users' data via GraphQL
Mass assignment     | Send extra fields: {"role":"admin"} in registration
```

# 6. API SECURITY TESTING
# ═══════════════════════════════════════════════════════════════════════════════

## 6.1 API Discovery

```
Step 1: Schema discovery
  - /swagger-ui/, /api-docs, /openapi.json, /swagger.json
  - /graphql (introspection query)
  - /api/, /v1/, /v2/ — version enumeration
  - WSDL for SOAP services

Step 2: Endpoint enumeration
  - From discovered schema
  - From JavaScript source code analysis
  - From mobile app analysis (if applicable)
  - Wordlist brute-force: /api/[FUZZ]

Step 3: Method testing
  - For each endpoint, try all HTTP methods
  - GET, POST, PUT, PATCH, DELETE, OPTIONS, HEAD
  - Note which methods are allowed vs blocked
```

## 6.2 API-Specific Vulnerabilities

```
BOLA (Broken Object-Level Authorization):
  - Change resource IDs in API calls
  - Try sequential IDs, UUIDs from other users
  - Test both read and write operations

Mass Assignment:
  - Send additional fields in create/update requests
  - Try: role, isAdmin, isPremium, balance, verified

Rate Limiting:
  - Test all sensitive endpoints for rate limits
  - Login, password reset, OTP verification, API keys

Excessive Data Exposure:
  - Compare API response to what the UI shows
  - API often returns more fields than necessary

Injection in API:
  - JSON injection in body parameters
  - SQL/NoSQL injection via API parameters
  - GraphQL injection and query batching
```

## 6.3 GraphQL-Specific Testing

```
Introspection:
  query { __schema { types { name fields { name type { name } } } } }
  → If enabled: full API schema exposed

Batching attack:
  [{"query":"mutation{login(user:\"A\",pass:\"1\")}{token}"},
   {"query":"mutation{login(user:\"A\",pass:\"2\")}{token}"},
   ...]
  → Bypass rate limiting on mutations

Depth/complexity attack:
  Deeply nested queries to cause DoS:
  { user { friends { friends { friends { ... } } } } }

Field suggestion:
  Try intentional typos to trigger "did you mean" suggestions
  → Enumerate hidden fields
```

# 7. CLIENT-SIDE VULNERABILITIES
# ═══════════════════════════════════════════════════════════════════════════════

## 7.1 Cross-Site Scripting (XSS)

```
Types and testing approach:

Reflected XSS:
  - Inject payloads in all parameters
  - Check if reflected in response without encoding
  - Test in different contexts (HTML, attribute, JS, URL)
  - Bypass filters: encoding, case variation, alternative tags

Stored XSS:
  - Input stored content (profiles, comments, messages)
  - Check where stored content is rendered
  - Test with other users' sessions viewing the stored content
  - Pay attention to admin panels rendering user input

DOM-based XSS:
  - Analyze JavaScript for dangerous sinks:
    document.write(), innerHTML, eval(), setTimeout()
  - Trace data flow from sources (location.hash, postMessage)
  - Test URL fragment payloads: #<img src=x onerror=alert(1)>
```

## 7.2 CSRF (Cross-Site Request Forgery)

```
Testing approach:
  1. Identify state-changing requests (POST, PUT, DELETE)
  2. Check for CSRF tokens in forms/requests
  3. Test token validation:
     - Remove token entirely
     - Use token from different session
     - Use empty token value
     - Modify token (change one character)
  4. Check SameSite cookie attribute
  5. Check CORS configuration
  6. Determine if custom headers are required (API-based CSRF defense)
```

## 7.3 Secrets in JavaScript

```
Search JavaScript source for:
  - API keys: /[A-Za-z0-9]{32,}/
  - AWS keys: AKIA[0-9A-Z]{16}
  - JWT tokens: eyJ[A-Za-z0-9-_]+\.eyJ[A-Za-z0-9-_]+
  - Private keys: -----BEGIN (RSA )?PRIVATE KEY-----
  - Internal URLs: https?://internal\.|https?://10\.|https?://192\.168
  - Endpoints: /api/admin, /internal, /debug
  - Hardcoded passwords: password\s*[:=]\s*['"][^'"]+

Tools:
  - LinkFinder: extract endpoints from JS
  - SecretFinder: extract secrets from JS
  - Manual grep with regex patterns
```

## 8.1 WordPress

```
Priority checks:
  1. Version detection: /readme.html, /wp-links-opml.php, meta generator
  2. Plugin enumeration: /wp-content/plugins/[FUZZ]/readme.txt
  3. Theme enumeration: /wp-content/themes/[FUZZ]/style.css
  4. User enumeration: /?author=1, /wp-json/wp/v2/users
  5. xmlrpc.php: system.multicall for brute force
  6. wp-cron.php: DoS amplification
  7. File upload via media: unrestricted file types?
  8. REST API exposure: /wp-json/ (user data, post data)

High-value findings:
  - Outdated plugins (especially: Contact Form 7, Elementor, WooCommerce)
  - xmlrpc.php enabled (brute force amplification)
  - Exposed wp-config.php backup (.bak, .old, ~)
  - Debug mode enabled (WP_DEBUG = true)
```

```
Drupal:
  - /CHANGELOG.txt (version)
```

# 9. 403 BYPASS TECHNIQUES
# ═══════════════════════════════════════════════════════════════════════════════

When encountering 403 Forbidden on interesting paths:

```
Technique                          | Example
-----------------------------------|----------------------------------
URL encoding                       | /admin → /%61%64%6d%69%6e
Double URL encoding                | /admin → /%2561%2564%256d%2569%256e
Path traversal                     | /./admin/, /../admin/
Trailing characters                | /admin/, /admin..;/, /admin;/
Case variation                     | /Admin, /ADMIN, /aDmIn
HTTP method change                 | GET → POST, HEAD, CONNECT
IP-based headers                   | X-Forwarded-For: 127.0.0.1
                                   | X-Original-URL: /admin
                                   | X-Rewrite-URL: /admin
Protocol downgrade                 | HTTPS → HTTP
HTTP version change                | HTTP/1.0 vs HTTP/1.1 vs HTTP/2
Null byte                          | /admin%00
Fragment/semicolon                 | /admin#, /admin;
Verb tampering                     | Custom method: GETT /admin
```
