# 3. SERVICE-SPECIFIC TESTING PLAYBOOKS
# ═══════════════════════════════════════════════════════════════════════════════

## 3.1 SSH (Port 22)

```
Enumeration:
  - Banner grab: ssh version, OS hints
  - Algorithm enumeration: nmap --script ssh2-enum-algos
  - Auth methods: nmap --script ssh-auth-methods
  - Host key fingerprint (compare across hosts for shared keys)

Vulnerabilities:
  - Outdated version with known CVEs
  - Weak algorithms (CBC ciphers, SHA-1 MACs, RSA < 2048)
  - Password auth enabled (should be key-only)
  - Known default keys (Raspberry Pi, embedded devices)
  - libssh auth bypass (CVE-2018-10933)

Testing:
  - Check auth methods: password, publickey, keyboard-interactive
  - Test common default credentials (if password auth enabled):
    root:root, admin:admin, pi:raspberry, ubuntu:ubuntu
  - Rate limit check on auth attempts
  - SFTP access if SSH auth succeeds
```

## 3.2 SMB (Ports 139, 445)

```
Enumeration:
  - nmap --script smb-os-discovery [target]
  - nmap --script smb-enum-shares [target]
  - nmap --script smb-enum-users [target]
  - smbclient -L [target] -N (null session)
  - crackmapexec smb [target]

Vulnerabilities:
  - SMBv1 enabled (EternalBlue, WannaCry — MS17-010)
  - Null session access (shares, user enum)
  - Guest access to shares
  - Writable shares (potential for malicious file placement)
  - SMB signing disabled (relay attacks possible)
  - PrintNightmare (CVE-2021-34527)

Testing:
  - Anonymous/Guest enumeration: smbclient -L //[target] -N
  - Share access test: smbmap -H [target]
  - Check for sensitive files in accessible shares
  - SMB signing status: nmap --script smb2-security-mode
  - Version check against known CVEs
```

## 3.3 FTP (Port 21)

```
Enumeration:
  - Banner grab (FTP server name and version)
  - nmap --script ftp-anon [target]
  - nmap --script ftp-syst [target]

Vulnerabilities:
  - Anonymous login permitted
  - Writable anonymous directory (upload malicious files)
  - Bounce attack (FTP PORT command abuse)
  - Cleartext protocol (credentials visible on wire)
  - Version-specific CVEs (vsftpd backdoor, ProFTPD exploits)

Testing:
  - Anonymous login: ftp [target] → user: anonymous, pass: empty
  - Directory listing and file access
  - Upload capability test (non-destructive — upload a test .txt file)
  - PASV mode vs PORT mode behavior
  - TLS/SSL support (AUTH TLS, FTPS)
```

## 3.4 RDP (Port 3389)

```
Enumeration:
  - nmap --script rdp-ntlm-info [target]
  - nmap --script rdp-enum-encryption [target]
  - OS and domain name from NTLM info

Vulnerabilities:
  - BlueKeep (CVE-2019-0708) — pre-auth RCE
  - NLA (Network Level Authentication) disabled
  - Weak encryption (40-bit, SSL, no NLA)
  - Brute-force susceptibility (no account lockout)
  - RDP session hijacking potential

Testing:
  - Check NLA requirement
  - Encryption level assessment
  - Certificate validation (self-signed? expired?)
  - BlueKeep detection: nmap --script rdp-vuln-ms12-020 (safe check)
  - Do NOT attempt actual BlueKeep exploitation (crash risk)
```

## 3.5 Database Services

```
MySQL (3306):
  - Test: mysql -h [target] -u root -p (empty password)
  - nmap --script mysql-info, mysql-enum
  - Check for remote root access
  - Default credentials: root:(empty), root:root, root:mysql

PostgreSQL (5432):
  - Test: psql -h [target] -U postgres
  - Default credentials: postgres:postgres
  - pg_hba.conf trust authentication
  - Check for default DB and schema access

MSSQL (1433):
  - nmap --script ms-sql-info, ms-sql-ntlm-info
  - Default: sa:(empty), sa:sa, sa:password
  - xp_cmdshell enabled? (RCE via SQL)
  - Linked servers (lateral movement)

Redis (6379):
  - Test: redis-cli -h [target] info
  - No authentication (common!)
  - If accessible: CONFIG GET dir, CONFIG GET dbfilename
  - Key enumeration: KEYS *
  - Write capability → potential RCE (cron, SSH keys, webshell)

MongoDB (27017):
  - Test: mongosh --host [target]
  - No authentication (historically very common)
  - Database enumeration: show dbs
  - Data exposure assessment

Elasticsearch (9200):
  - Test: curl http://[target]:9200/
  - curl http://[target]:9200/_cat/indices
  - No authentication (common)
  - Index data exposure
  - Kibana dashboard (5601)
```

## 3.6 SNMP (UDP 161)

```
Enumeration:
  - snmpwalk -v2c -c public [target]
  - Test community strings: public, private, community
  - nmap --script snmp-info, snmp-brute

Findings:
  - Default community strings (public read, private write)
  - System information disclosure (OS, hostname, interfaces)
  - SNMPv1/v2c (no encryption, community string in cleartext)
  - Write access (change configurations!)
  - Network interface enumeration (discover hidden networks)
```

## 3.7 SMTP (Port 25, 587)

```
Enumeration:
  - Banner grab (MTA type and version)
  - nmap --script smtp-enum-users [target]
  - nmap --script smtp-open-relay [target]

Vulnerabilities:
  - Open relay (send email as anyone)
  - User enumeration (VRFY, EXPN, RCPT TO)
  - Missing STARTTLS
  - SPF/DKIM/DMARC not configured (email spoofing)
  - Version-specific CVEs (Exim, Postfix, Exchange)
```

## 3.8 DNS (Port 53)

```
Testing:
  - Zone transfer: dig AXFR [domain] @[target]
  - Version query: dig version.bind CHAOS TXT @[target]
  - Recursion enabled: dig +short test.openresolver.com @[target]
  - Cache poisoning susceptibility
  - DNSSEC validation

Findings:
  - Zone transfer permitted (complete domain dump)
  - Open resolver (DNS amplification attack vector)
  - Version disclosure
  - Missing DNSSEC
```

## 3.9 LDAP (Port 389, 636)

```
Enumeration:
  - ldapsearch -x -H ldap://[target] -b "" -s base
  - Anonymous bind test
  - Base DN discovery
  - User/group enumeration

Vulnerabilities:
  - Anonymous bind permitted
  - Cleartext LDAP (no TLS/STARTTLS on port 389)
  - User enumeration without authentication
  - Weak password policy
  - Interesting attributes exposure (passwords in description fields)
```

# 4. SSL/TLS CONFIGURATION AUDIT
# ═══════════════════════════════════════════════════════════════════════════════

## 4.1 Protocol Testing

```
Check for supported protocols:
  ✗ SSLv2    — Critically insecure, DROWN attack
  ✗ SSLv3    — POODLE attack
  ✗ TLS 1.0  — PCI DSS non-compliant since June 2018
  ✗ TLS 1.1  — Deprecated by all major browsers
  ✓ TLS 1.2  — Acceptable with good cipher suites
  ✓ TLS 1.3  — Recommended, strongest available

Tool: testssl.sh --protocols [target]
   or nmap --script ssl-enum-ciphers -p 443 [target]
```

## 4.2 Cipher Suite Analysis

```
Severity ratings for cipher issues:
  CRITICAL: Export ciphers (FREAK), NULL ciphers, RC4
  HIGH    : CBC mode ciphers (BEAST/Lucky13), 3DES (Sweet32)
  MEDIUM  : SHA-1 signatures, RSA key exchange (no PFS)
  LOW     : Missing AEAD ciphers, suboptimal ordering
  GOOD    : ECDHE/DHE + AES-GCM/ChaCha20 + SHA-256/384
```

## 4.3 Certificate Analysis

```
Check:
  - Expiration date (expired = warning shown to users)
  - Subject and SANs (additional domain names!)
  - Issuer (self-signed? unknown CA?)
  - Key size (RSA < 2048 or ECC < 256 is weak)
  - Signature algorithm (SHA-1 is deprecated)
  - Certificate transparency log inclusion
  - Revocation status (CRL/OCSP)
  - Wildcard certificate scope
```

## 4.4 Known TLS Vulnerabilities

```
Vulnerability    | Test                          | Severity
-----------------|-------------------------------|----------
Heartbleed       | nmap --script ssl-heartbleed  | Critical
POODLE           | SSLv3 + CBC ciphers enabled   | High
BEAST            | TLS 1.0 + CBC ciphers         | Medium
CRIME/BREACH     | TLS compression enabled       | Medium
ROBOT            | RSA key exchange issues        | High
Ticketbleed      | F5 session tickets             | Medium
CCS Injection    | OpenSSL CCS vulnerability     | High
```

# 5. DEFAULT CREDENTIAL TESTING
# ═══════════════════════════════════════════════════════════════════════════════

## 5.1 Methodology

```
IMPORTANT: Credential testing rules:
  1. Only test COMMON defaults — not brute-force
  2. Maximum 5-10 attempts per service
  3. Check for account lockout FIRST
  4. Stop after first successful login
  5. Document the finding, do NOT explore further without authorization
```

# 6. NETWORK SEGMENTATION ASSESSMENT
# ═══════════════════════════════════════════════════════════════════════════════

## 6.1 What to Look For

```
Segmentation indicators:
  - Can external-facing services reach internal networks?
  - Are management interfaces accessible from untrusted networks?
  - Do different environments (dev/staging/prod) share networks?
  - Are database ports accessible from the internet?
  - Can you reach cloud metadata from external-facing apps (SSRF)?

Trust relationships:
  - Shared credentials between services
  - Trust-based authentication (SSH keys, Kerberos, etc.)
  - Service accounts with broad access
  - Database links and federated queries
```

## 6.2 Lateral Movement Path Mapping

```
For each confirmed access:
  1. What network can I see from here?
  2. What services are accessible that weren't before?
  3. Are there credential reuse opportunities?
  4. Are there trust relationships to exploit?
  5. Map: [entry point] → [pivot] → [high-value target]
```

# 7. SAFETY GUIDELINES — SERVICE-SPECIFIC
# ═══════════════════════════════════════════════════════════════════════════════

```
Service    | Safe Actions                    | NEVER Do
-----------|---------------------------------|----------------------------
SMB        | Enumerate, read shares          | Write/delete files
FTP        | List, read files                | Upload to prod paths
SSH        | Auth test, config check         | Modify files/configs
Database   | Read-only queries               | DROP, DELETE, UPDATE, INSERT
RDP        | Check config, NLA status        | Login and interact
DNS        | Zone transfer, queries          | Modify DNS records
SNMP       | Read community string tests     | Write community actions
SMTP       | Relay test with own domain      | Send to real external users
Redis      | INFO, KEYS, read-only           | DEL, FLUSHALL, CONFIG SET
LDAP       | Read queries, enumeration       | Modify entries
```
