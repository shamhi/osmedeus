# 3. NUCLEI TEMPLATE STRATEGY
# ═══════════════════════════════════════════════════════════════════════════════

## 3.1 Tiered Scanning Approach

Run Nuclei in waves, not all at once.  Each wave informs the next.

```
Wave 1: Critical Quick Hits (run first, ~2-5 minutes)
  nuclei -u [target] -severity critical,high -type http \
         -tags cve,rce,auth-bypass,sqli -stats

  Purpose: Catch known critical CVEs and high-impact issues immediately.
  Action on findings: Validate and report immediately.

Wave 2: Misconfiguration & Exposure (~5-10 minutes)
  nuclei -u [target] -severity medium,high \
         -tags misconfig,exposure,default-login,token \
         -type http -stats

  Purpose: Find configuration errors, exposed panels, leaked credentials.
  Action on findings: Triage by exploitability.

Wave 3: Technology-Specific Templates (~5-15 minutes)
  Based on identified technology stack, run targeted templates:

  WordPress   : nuclei -tags wordpress
  Spring Boot : nuclei -tags springboot (actuators, SpEL injection)
  Apache      : nuclei -tags apache
  nginx       : nuclei -tags nginx
  PHP         : nuclei -tags php
  Node.js     : nuclei -tags nodejs
  .NET        : nuclei -tags dotnet
  Jenkins     : nuclei -tags jenkins
  Tomcat      : nuclei -tags tomcat

Wave 4: Broad Sweep (if time permits, ~10-30 minutes)
  nuclei -u [target] -severity low,medium,high,critical \
         -type http -stats -rate-limit 50

  Purpose: Catch anything missed by targeted waves.
  Action: Triage aggressively — expect noise.
```

## 3.2 Template Selection Heuristics

```
If technology is known:
  → Use technology-specific templates FIRST
  → Then general templates

If behind WAF/CDN:
  → Reduce rate (-rate-limit 10)
  → Skip brute-force templates
  → Focus on logic and config issues

If target is API:
  → Use API-specific templates
  → Focus on BOLA, injection, auth bypass

If target has authentication:
  → Test pre-auth endpoints first
  → Try default credentials
  → Then test with any discovered credentials
```

## 3.3 Custom Template Indicators

Write or modify Nuclei templates when:
  - A service has a known CVE but no existing template
  - You find a pattern that might be exploitable but isn't covered
  - The default template has too many false positives for this target
  - You need to chain multiple requests for validation

# 4. CVE ANALYSIS METHODOLOGY
# ═══════════════════════════════════════════════════════════════════════════════

## 4.1 Version-to-CVE Matching

For each identified service version:

```
Step 1: Exact version match
  - Search: "product version CVE"
  - Sources: NVD, CVE Details, vendor advisories

Step 2: Version range analysis
  - Many CVEs affect version ranges (e.g., "Apache < 2.4.50")
  - Determine if the discovered version falls within the range

Step 3: Exploit availability check
  - searchsploit [product] [version]
  - GitHub PoC search
  - Metasploit module check
  - Is there a working exploit? PoC? Just advisory?

Step 4: Exploitability assessment
  - CVSS exploitability metrics (attack vector, complexity, privileges)
  - Real-world exploitability (is it practical?)
  - Environmental factors (WAF, network position, auth requirements)
```

## 4.2 CVE Prioritization Matrix

```
Priority | CVSS  | Exploit Available | Attack Vector | Action
---------|-------|-------------------|---------------|------------------
P0       | 9.0+  | Yes (public)      | Network       | Validate NOW
P1       | 7.0+  | Yes (public)      | Network       | Validate soon
P2       | 7.0+  | No                | Network       | Assess feasibility
P3       | 4.0+  | Yes               | Any           | Note for chaining
P4       | <4.0  | Any               | Any           | Note in report
```

# 5. OWASP TOP 10 TESTING GUIDE
# ═══════════════════════════════════════════════════════════════════════════════

## A01: Broken Access Control
  Tests:
    - Modify user IDs in URLs/params (IDOR)
    - Access admin endpoints as regular user
    - Test HTTP method switching (GET→POST→PUT→DELETE)
    - Check CORS configuration (Access-Control-Allow-Origin: *)
    - Try accessing other users' data via API
  Indicators:
    - Different response for existing vs non-existing IDs (IDOR)
    - Missing authorization checks on state-changing operations
    - Overly permissive CORS

## A02: Cryptographic Failures
  Tests:
    - SSL/TLS configuration (protocols, ciphers)
    - Sensitive data in URLs (tokens, passwords)
    - Unencrypted HTTP for sensitive pages
    - Weak hashing algorithms in responses
    - Exposed secrets in JavaScript, .env, config files
  Indicators:
    - TLS 1.0/1.1, weak ciphers, expired certificates
    - Sensitive data visible in plain text

## A03: Injection
  Tests:
    - SQL injection: ' OR 1=1--, UNION SELECT, blind with time delays
    - NoSQL injection: {$gt:""}, {$regex:".*"}
    - OS command injection: ;id, |whoami, `id`, $(id)
    - SSTI: {{7*7}}, ${7*7}, #{7*7}, <% 7*7 %>
    - LDAP injection: *, )(cn=*), etc.
    - XPath injection
  Indicators:
    - Error messages revealing SQL/NoSQL syntax
    - Time-based responses for blind injection
    - Reflected calculations in SSTI

## A05: Security Misconfiguration
  Tests:
    - Default credentials on admin panels
    - Verbose error messages (stack traces, SQL errors)
    - Directory listing enabled
    - Unnecessary HTTP methods (TRACE, OPTIONS)
    - Missing security headers
    - Exposed management interfaces (phpMyAdmin, Adminer, etc.)
    - Debug mode enabled (?debug=true, /_debug/, etc.)
  Indicators:
    - Detailed error responses
    - Default pages (Apache/nginx/IIS default)
    - Accessible admin interfaces

## A07: Identification and Authentication Failures
  Tests:
    - Brute force protection (rate limiting, account lockout)
    - Password policy (minimum length, complexity)
    - Default credentials
    - Session management (cookie flags, session fixation)
    - Password reset flow vulnerabilities
    - JWT weaknesses (none algorithm, weak secret, missing validation)
  Indicators:
    - No rate limiting on login
    - Session tokens in URL
    - JWT with alg:none or weak HMAC secret

## A10: Server-Side Request Forgery (SSRF)
  Tests:
    - URL parameters that fetch external resources
    - File upload with URL input
    - Webhook/callback URLs
    - PDF generators with URL input
    - Try: http://127.0.0.1, http://169.254.169.254, http://[::1]
  Indicators:
    - Different response times for internal vs external URLs
    - Error messages revealing internal network info
    - Successful access to metadata endpoints

# 6. LOGIC VULNERABILITIES — THE SCANNER BLIND SPOT
# ═══════════════════════════════════════════════════════════════════════════════

Automated scanners miss logic bugs.  This is where your human-like reasoning
adds the most value.

## 6.1 Common Logic Vulnerability Patterns

```
Race Conditions:
  - Double-spend in payment flows
  - TOCTOU in file operations
  - Parallel requests to bypass limits
  Test: Send identical requests simultaneously

Price/Quantity Manipulation:
  - Negative quantities, zero prices, overflow values
  - Currency conversion inconsistencies
  Test: Modify hidden parameters in purchase flows

Workflow Bypass:
  - Skip steps in multi-step processes
  - Direct access to completion endpoints
  Test: Jump directly to step 3 of a 5-step process

Role/Permission Confusion:
  - Privilege escalation through parameter manipulation
  - Cross-tenant data access in multi-tenant apps
  Test: Swap user/org IDs between accounts

Reset/Recovery Abuse:
  - Password reset token predictability
  - No rate limit on reset requests
  - Reset token reuse
  Test: Analyze token entropy, test for reuse
```

## 6.2 How to Find Logic Bugs

1. Map the application's business logic flow
2. Identify trust assumptions (what does the app take for granted?)
3. Violate those assumptions systematically
4. Focus on state-changing operations (payments, auth, data modification)
5. Test edge cases: empty, null, negative, max int, special characters

# 7. FALSE POSITIVE IDENTIFICATION
# ═══════════════════════════════════════════════════════════════════════════════

## 7.1 Common False Positive Patterns

```
Pattern                          | Likely False Positive?
---------------------------------|------------------------
Same vuln on every endpoint      | Yes — WAF/CDN interference
Generic Nuclei info template hit | Maybe — verify manually
Version-based CVE, no PoC works  | Likely — patched but banner unchanged
XSS in parameter that's not      |
  rendered in response            | Yes — input sanitized server-side
SQL injection with no observable  |
  behavior change                 | Maybe — blind SQLi or FP
```

## 7.2 Validation Steps

For every automated finding:
  1. Can I reproduce it manually?
  2. Does the response actually indicate the vulnerability?
  3. Is this a real server response or a WAF/CDN page?
  4. Is the "vulnerable" version actually patched (backported fix)?
  5. Does the application's security controls mitigate this?

Rate your confidence:
  - CONFIRMED: Manually validated, clear evidence
  - PROBABLE: Strong indicators, high confidence but not 100%
  - POSSIBLE: Some evidence, needs more investigation
  - LIKELY FP: Indicators suggest false positive, deprioritize

# 8. ATTACK CHAIN CONSTRUCTION
# ═══════════════════════════════════════════════════════════════════════════════

After individual vulnerability discovery, look for chains:

## 8.1 Common Chain Patterns

```
Chain: Info Disclosure → Account Takeover
  Step 1: Discover user enumeration endpoint
  Step 2: Find password reset with predictable token
  Step 3: Enumerate users + generate reset tokens
  Impact: Mass account compromise

Chain: SSRF → Cloud Compromise
  Step 1: Find SSRF endpoint
  Step 2: Access cloud metadata (169.254.169.254)
  Step 3: Steal IAM credentials
  Impact: Full cloud infrastructure compromise

Chain: Open Redirect → OAuth Token Theft
  Step 1: Find open redirect on the OAuth client domain
  Step 2: Craft malicious OAuth flow with redirect
  Step 3: Steal authorization code or token
  Impact: Account takeover via OAuth

Chain: XSS → Session Hijack → Admin Access
  Step 1: Find stored XSS in user-controlled content
  Step 2: Craft payload to steal admin session
  Step 3: Use admin session for privilege escalation
  Impact: Full application compromise
```

## 8.2 Chain Building Process

1. List all confirmed and probable findings
2. For each finding, ask: "What can I reach from here?"
3. Map findings to MITRE ATT&CK techniques
4. Draw edges between findings that enable each other
5. Calculate combined impact (often > sum of parts)
6. Prioritize chains by: impact × feasibility
