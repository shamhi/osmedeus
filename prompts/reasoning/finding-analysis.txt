After receiving tool output, work through this analysis template before
proceeding to the next action.  Rushing past results is how real attackers
miss critical findings and how automated scanners generate noise.

# STEP 1: RAW OUTPUT PARSING
# ═══════════════════════════════════════════════════════════════════════════════

Systematically extract information from the raw output:

```
RAW OUTPUT ANALYSIS
====================
Tool Used     : [tool name and exact command]
Target        : [what was scanned/tested]
Execution Time: [how long it took — unusually fast or slow?]
Exit Status   : [success, error, timeout, partial]
Output Size   : [amount of data returned — expected or surprising?]

Extracted Data Points:
  1. [Data point: value — significance]
  2. [Data point: value — significance]
  3. [Data point: value — significance]
  ...

Data Quality Assessment:
  - Completeness: [Did the tool finish? Any truncation?]
  - Reliability : [Is this tool generally accurate for this type of data?]
  - Freshness   : [Are these current results or could they be cached/stale?]
```

## 1.1 Tool-Specific Parsing Guides

### Nmap Output Analysis
```
For each discovered service, extract:
  □ Port number and protocol (TCP/UDP)
  □ Service name (what nmap thinks it is)
  □ Version (exact version string)
  □ State (open/filtered/closed — filtered is interesting!)
  □ NSE script output (often contains vulns)
  □ OS detection results
  □ Traceroute data (network path info)

Red flags in nmap output:
  - "tcpwrapped" — service is hiding behind TCP wrapper
  - Many filtered ports — firewall is active
  - Unusual ports with known services — deliberate hiding?
  - Version strings that look custom or modified
```

### Nuclei Output Analysis
```
For each finding, extract:
  □ Template ID and name
  □ Severity level
  □ Matched URL/endpoint
  □ Matched string/pattern
  □ Confidence indicator
  □ CVE reference (if applicable)

Red flags in Nuclei output:
  - Same template matching every endpoint — likely FP (WAF page)
  - Info-level findings that reveal versions — feed into CVE matching
  - "exposure" findings — check if data is actually sensitive
  - Technology detection — update your tech stack profile
```

### HTTP Response Analysis
```
For each HTTP response, extract:
  □ Status code and reason
  □ Response headers (all of them)
  □ Set-Cookie attributes
  □ Content-Type and encoding
  □ Body content (relevant portions)
  □ Redirect chain (if 3xx)
  □ Response timing

Red flags in HTTP responses:
  - Verbose error messages (stack traces, SQL errors)
  - Internal IP addresses or hostnames leaked
  - Unexpected headers (X-Debug, X-Backend-Server)
  - Different response size/timing for similar requests (timing oracle)
  - Set-Cookie without Secure/HttpOnly/SameSite flags
```

### Directory Discovery Output Analysis
```
For each discovered path, extract:
  □ Path and status code
  □ Response size (helps identify unique pages vs default)
  □ Content type
  □ Redirect destination (if 3xx)
  □ Authentication requirement (if 401/403)

Red flags in discovery output:
  - Admin/management paths (even if 403 — attempt bypass)
  - Backup/archive files (.bak, .old, .zip, .sql)
  - Source code or config files (.env, .git, web.config)
  - API documentation paths (/swagger, /api-docs)
  - Debug/diagnostic endpoints (/debug, /trace, /health)
  - Status code anomalies (one path returns 500 — possible injection)
```

# STEP 2: TRUE POSITIVE vs FALSE POSITIVE ASSESSMENT
# ═══════════════════════════════════════════════════════════════════════════════

For each potential finding, assess whether it's real:

```
FINDING TRIAGE
===============
Finding: [brief description]
Source : [which tool reported it]

True Positive Indicators:
  □ Specific, unique response that matches the vulnerability
  □ Behavior change when payload is modified
  □ Consistent reproduction across multiple attempts
  □ Tool-specific confidence is high
  □ Matches known vulnerability pattern for this technology

False Positive Indicators:
  □ Same finding reported for every endpoint (generic match)
  □ Response is a WAF/CDN error page, not the actual server
  □ Version-based match but software may be patched (backports)
  □ Template matches on HTML comment or non-functional context
  □ No behavioral difference between normal and attack requests

Verdict: [CONFIRMED | PROBABLE | POSSIBLE | LIKELY_FP | FP]
Confidence: [1-10]
Action: [Validate further | Report as-is | Investigate deeper | Discard]
```

## 2.1 False Positive Decision Tree

```
Is the same finding on every tested endpoint?
  YES → Almost certainly FP (WAF/CDN generic response)
  NO  ↓

Does the response contain the actual vulnerable behavior?
  YES → Likely TP — validate further
  NO  ↓

Is this a version-based detection only?
  YES → Check if version is accurate (not just banner)
        Check if patching without version bump is common
  NO  ↓

Does the finding require specific conditions that may not exist?
  YES → Mark as POSSIBLE, note conditions needed
  NO  ↓

Can you reproduce the finding manually?
  YES → CONFIRMED
  NO  → Likely FP — investigate why tool flagged it
```

# ═══════════════════════════════════════════════════════════════════════════════
# STEP 5: PATTERN RECOGNITION
# ═══════════════════════════════════════════════════════════════════════════════

Look for patterns across findings — individual data points become insights
when connected:

```
PATTERN ANALYSIS
=================

Cross-Finding Patterns:
  - Are multiple findings related to the same root cause?
    (e.g., multiple injection points = no input validation framework)
  - Are findings clustered on specific hosts or applications?
    (e.g., staging server has more vulns = less security attention)
  - Do findings follow a technology-specific pattern?
    (e.g., all Java deserialization points = systemic design issue)

Absence Patterns (what's NOT there):
  - Expected service that's missing? (might be filtered, not absent)
  - Common vulnerability not found? (good security or missed test?)
  - No error messages at all? (aggressive error handling — try harder)

Temporal Patterns:
  - Response times varying? (could indicate backend differences)
  - Intermittent results? (load balancer with different configs)
  - Scan results differ from earlier? (dynamic infrastructure)

Attack Surface Patterns:
  - Consistent security posture or inconsistent?
  - Old and new systems mixed? (old = likely weaker)
  - Different teams managing different parts? (inconsistent config)
```

## Example: Analyzing Nuclei Output

```
RAW OUTPUT:
  [critical] CVE-2021-44228 [apache-log4j-rce] [http]
    → https://app.example.com/api/login
  [high] springboot-actuator [exposure] [http]
    → https://app.example.com/actuator/env
  [medium] missing-x-frame-options [misconfiguration] [http]
    → https://app.example.com
  [info] waf-detect [tech] [http]
    → https://app.example.com [cloudflare]

ANALYSIS:

1. CVE-2021-44228 (Log4Shell) — NEEDS VALIDATION
   The target runs Java (Spring Boot confirmed by actuator finding).
   Log4j RCE is critical if real.  However:
   - Cloudflare WAF might be triggering on the test payload
   - Need to verify with a safe payload (DNS-based callback)
   - If behind Cloudflare, the actual Log4j test might not reach
     the application
   Action: Validate with DNS callback payload to confirm execution

2. Spring Boot Actuator Exposure — LIKELY TP
   Actuator /env endpoint exposed = confirmed Spring Boot.
   This endpoint leaks environment variables including potential:
   - Database credentials
   - API keys
   - Internal URLs
   Action: Read actuator/env output, check for sensitive data.
   Also check: /actuator/health, /actuator/mappings, /actuator/beans

3. Missing X-Frame-Options — TP (LOW SEVERITY)
   Standard misconfiguration finding.  Low severity unless:
   - Combined with XSS for clickjacking attack
   - Sensitive actions on the page (payments, settings)
   Action: Note for report, low priority

4. Cloudflare WAF Detected — INFORMATIONAL
   Important context for all other testing:
   - Reduce scan intensity
   - Some findings might be WAF-generated FPs
   - Consider finding origin IP
   Action: Update assessment context, adjust scanning approach

PATTERN: Java/Spring application behind Cloudflare with exposed
management endpoints.  This suggests security oversight on the
application layer despite having WAF protection.

CHAIN POTENTIAL: Actuator env → credentials → deeper access
                 OR Log4Shell (if real) → immediate RCE
```
