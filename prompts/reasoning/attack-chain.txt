## 3.1 Common Chain Patterns to Look For

```
Pattern: Info Leak → Credential Access → Full Compromise
  Finding A: Information disclosure (internal IPs, usernames, versions)
  Finding B: Credential exposure (hardcoded creds, weak passwords)
  Finding C: Service access using discovered credentials
  Impact escalation: Info (Low) → Creds (Medium) → Access (Critical)

Pattern: SSRF → Internal Service → Data/RCE
  Finding A: SSRF in web application
  Finding B: Internal service with no auth (Redis, metadata, admin panel)
  Finding C: Ability to read data or execute commands via internal service
  Impact escalation: SSRF (Medium) → Internal (High) → RCE (Critical)

Pattern: XSS → Session Theft → Account Takeover
  Finding A: Stored XSS in user-controlled content
  Finding B: Session cookies without HttpOnly flag
  Finding C: Session hijacking of other users (including admins)
  Impact escalation: XSS (Medium) → Session (High) → Admin (Critical)

Pattern: Auth Bypass → Privilege Escalation → Data Access
  Finding A: Authentication bypass on specific endpoint
  Finding B: Privilege escalation via parameter manipulation
  Finding C: Access to sensitive data with elevated privileges
  Impact escalation: Bypass (High) → Escalation (High) → Data (Critical)

Pattern: Subdomain Takeover → Cookie Scoping → Session Hijack
  Finding A: Dangling CNAME (subdomain takeover possible)
  Finding B: Cookies scoped to parent domain (not specific subdomain)
  Finding C: Attacker controls subdomain, can read parent domain cookies
  Impact escalation: Takeover (Medium) → Cookie (Medium) → Hijack (Critical)

Pattern: Open Redirect → OAuth Flow → Token Theft
  Finding A: Open redirect on OAuth client domain
  Finding B: Redirect_uri validation allows the open redirect URL
  Finding C: Authorization code/token sent to attacker via redirect
  Impact escalation: Redirect (Low) → OAuth (High) → Token (Critical)

Pattern: File Read → Source Code → Hardcoded Secrets → Access
  Finding A: Local file read (LFI, path traversal)
  Finding B: Application source code accessible
  Finding C: Hardcoded credentials in source code
  Finding D: Credentials work on production services
  Impact escalation: LFI (High) → Source (High) → Creds (Critical)

Pattern: API IDOR → Mass Data Access → PII Exposure
  Finding A: IDOR on user data API endpoint
  Finding B: Sequential/predictable user IDs
  Finding C: Automated enumeration of all user records
  Impact escalation: IDOR (Medium) → Enumeration (High) → PII (Critical)
```

  P0 = Critical impact + High feasibility + Validated
  P1 = High impact + High feasibility + Validated

# COMPLETE CHAIN ANALYSIS EXAMPLE
# ═══════════════════════════════════════════════════════════════════════════════

```
ATTACK CHAIN: Customer Database Compromise via Application Stack

Findings Used:
  F-003: Spring Boot Actuator exposed at /actuator/env (Medium)
  F-007: Database credentials in actuator environment (High)
  F-012: Database port (5432) accessible from internet (Medium)

Step 1: Discovery — Actuator Endpoint Exposure
  Technique: T1190 (Exploit Public-Facing Application)
  Action: Access https://app.example.com/actuator/env
  Result: Environment variables exposed including SPRING_DATASOURCE_URL,
          SPRING_DATASOURCE_USERNAME, SPRING_DATASOURCE_PASSWORD
  Evidence: HTTP 200 response with full env dump

Step 2: Credential Extraction — Database Credentials
  Technique: T1552.001 (Credentials in Files)
  Action: Parse actuator output for database credentials
  Result: PostgreSQL credentials: dbuser / [REDACTED]
          Database host: db.internal.example.com (resolves to 10.0.1.50,
          but CNAME also points to rds.amazonaws.com — internet accessible)
  Evidence: Credentials visible in plain text in actuator response

Step 3: Data Access — Direct Database Connection
  Technique: T1078 (Valid Accounts)
  Action: Connect to database with discovered credentials
  Result: Full read access to production database containing:
          - 150,000 user records (name, email, hashed passwords)
          - 50,000 order records (payment info partially tokenized)
          - Application configuration including API keys
  Evidence: Successful psql connection and SELECT count(*)

Chain Visualization:
  [Actuator Exposed] ──→ [DB Creds Leaked] ──→ [Database Access]
    (Medium 5.3)           (High 7.5)           (Critical 9.1)
         └────────── Combined: CRITICAL (9.8) ──────────┘

Blast Radius:
  - Direct: 150,000 user records, payment data, application secrets
  - Indirect: GDPR/CCPA notification required, potential class action
  - Regulatory: PCI-DSS violation (database credentials in plain text)

Break Points:
  1. Disable actuator endpoints in production (Easy — config change)
  2. Use secret management (Vault/SSM) instead of env vars (Medium)
  3. Restrict database access to application VPC only (Medium)
  4. Use IAM auth for database instead of passwords (Hard)

Recommendation: Fix #1 immediately (15 minutes), schedule #2 and #3
                for next sprint, plan #4 for architecture roadmap.
```
